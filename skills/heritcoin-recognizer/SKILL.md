# Heritcoin Recognizer

用于识别、鉴定、估价单枚硬币。一次任务只处理同一枚钱币的 2 张图片，并且只调用一次识别脚本。

## 何时使用

- 用户提供同一枚钱币的图片引用。
- 用户想知道这是什么币、年份、国家或地区、材质、价格等。
- 任务目标是识别单枚钱币，不是批量识别多枚钱币。

## 核心原则

- 一次识别任务 = 同一枚钱币的 2 张图片。
- 不要直接用模型原生视觉回答，必须走脚本。
- 不要把每张图单独识别，也不要把一次任务拆成多次调用。
- 当前任务只有 1 张图时，只能提示还缺同一枚钱币的另一张图；不得直接用模型原生视觉猜测币种、年份、国家或估价。
- 超过 2 张图时直接拒绝：`一次最多上传2张图，请重新上传`
- 聊天附件恢复属于独立桥接层，不属于 `recognize.ts` 的职责。

## 环境准备

首次使用前，必须在 `scripts/` 目录安装依赖：

```bash
cd scripts && npm install
```

## 最小工作流

1. 收集当前任务图片  
   支持：
- 聊天附件图片
- 图片 URL
- 本地图片路径
- `data:image/...;base64,...`

2. 如果当前任务包含聊天附件，先运行桥接脚本  
   在 `scripts/` 目录运行：

```bash
npx tsx resolve-image-inputs.ts
```

- 桥接脚本负责从当前宿主线程恢复这次任务的图片输入。
- 它会同时考虑聊天附件、显式图片 URL、本地图片路径，以及“还差一张图”后的补图消息。
- 它不会调用识别服务，只输出当前任务的 `images` 数组。
- 如果桥接脚本缺少当前线程上下文，它应当直接失败，而不是回退到别的会话。

3. 按当前任务图片数决定动作
- `0` 张：提示用户上传同一枚钱币的 2 张图片，并明确收到 2 张后即可继续识别和估价。
- `1` 张：明确当前只收到 1 张，还缺同一枚钱币的另一张图；同时明确用户补齐后即可继续识别和估价；不要调用识别，也不要直接用模型原生视觉给出识别结论。
- `2` 张：调用识别脚本 1 次。
- `>2` 张：直接回复 `一次最多上传2张图，请重新上传`，不要调用识别。

4. 只使用这个识别入口  
   在 `scripts/` 目录运行：

```bash
npx tsx recognize.ts <img1> <img2> [--token <token>] [--locale <locale>]
```

- 除非用户明确提供 `token`，否则使用脚本自身认证配置。
- 用户语言很明确时，显式传入对应 `--locale`。
- 没有显式传入 `--locale`` 时，脚本只回退到系统 locale。

5. 最终输出契约

- 脚本成功时，最终回复必须原样返回脚本生成的 Markdown 最终识别结果；不要附加“上传中”“上传完成”等过程日志。
- agent 不得改写、压缩、摘要、重排，也不得省略任何非空字段。
- 主体使用 Markdown 标题加两列表格。
- “名称/币名”和“估价”必须排在最前面。
- 收藏建议放在表格后，单独一行输出。
- 不要使用 Markdown 代码块包裹最终结果。

## 用户交互约束

- 用户一次发来 2 张图时，直接识别 1 次。
- 用户先发 1 张，后续补第 2 张时，在第 2 张到达后识别 1 次。
- 用户只发 1 张图时，要明确告诉用户补齐同一枚钱币的另一张图后，就可以继续识别和估价；不要直接根据当前这 1 张图使用模型原生视觉回答“这是什么币”。
- assistant 明确催补图后，只有紧接着的下一条用户消息补发同一枚钱币的单张图片引用时，才自动把它并入当前未完成任务。
- 如果用户只是单独发 1 个新的图片引用，且没有补图语义，也没有刚收到 assistant 的补图提示，默认把它当成新任务第 1 张图。
- 已经凑齐 2 张图后，不要再说“还差一张”。
- 不要在同一次任务中重复调用脚本。
- 不要引用已完成旧任务里的图片或结果。
- 不要把脚本返回的完整结果改写成更短摘要，导致材质、直径、厚度、重量等非空字段丢失。
- 用户当前请求是中文时，不要输出英文结果；除非用户明确要求英文。

## 结果字段要求

- 至少保留：币名、估价、年份、国家或地区、面值。
- 如果脚本返回了 Krause 编号、材质、直径、厚度、重量、正面描述、背面描述或其它补充字段，也必须保留。
