# Heritcoin Recognizer

## 目标

将一次识别请求视为“同一枚钱币的一个待处理任务”。仅在收集到同一枚钱币的 2 张图片后执行 1 次识别。

## 执行规则

1. 先收集当前识别任务中的全部图片输入。
   支持：
   - 图片 URL
   - 本地图片路径
   - 聊天附件图片
   当前消息里出现的显式图片 URL / 本地图片路径，也属于当前任务图片输入。
   对聊天附件，不要直接看图作答；先在 `scripts/` 目录运行：

   ```bash
   node resolve-chat-images.ts
   ```

   该脚本会从当前 Codex session 日志里提取最近待处理任务的附件图片，并将附件落成临时文件路径。把它返回的 `images` 数组视为当前任务的可执行图片输入。

2. 再合并最近一个未完成任务中的图片。
   如果最近一个待处理任务已经收集了 1 张同一枚钱币的图片，将本条消息中的图片并入该任务。
   如果当前消息没有新图片，但最近一个待处理任务已经收集到 2 张同一枚钱币的图片，而用户当前是在追问“这是什么币”“帮我识别”“值多少钱”这类识别意图，也要继续使用该任务。
   如果用户当前消息提供的是显式图片 URL / 本地路径，只有在以下两种情况下才自动并入最近未完成任务：
   - assistant 紧接着上一条最终回复刚刚明确要求“还差一张”“请补另一面/第二张”
   - 用户当前文字明确说明“补第二张/另一面/同一枚”
   除了这两种情况，单张显式 URL / 本地路径默认视为新任务的第 1 张图，不要因为历史里还有未完成任务就自动凑成 2 张并识别。
   assistant 已经给出识别结果或其它最终答复后，该任务就视为已完成；后续新图片必须开启新任务，不能回收更早历史图片。
   不要把每张图片单独当成一次识别。

3. 只按当前识别任务中的图片总数决定下一步。
   - 总数为 0：提示用户上传同一枚钱币的 2 张图片。
   - 总数为 1：明确只缺 1 张图片，不要调用识别。
   - 总数为 2：调用脚本 1 次并返回 1 份整合结果。
   - 总数大于 2：直接拒绝处理，不要调用识别；固定提示“`一次最多上传2张图，请重新上传`”。

4. 不要把“是否正反面”当成调用前置条件。
   识别接口只要求 2 张同一枚钱币的图片。
   如果用户说明了正反面，按该顺序传入脚本。
   如果用户未说明但已经提供了 2 张图，直接将这 2 张图作为同一次识别输入。
   不要在已经拿到 2 张图后再次要求补图。

5. 只使用一个执行入口。
   在 `scripts/` 目录运行：

   ```bash
   node recognize.ts <img1> <img2> [token] [locale]
   ```

   不要执行 `npm install`，当前脚本应直接由 Node 运行。
   除非用户明确提供 `token`，否则让脚本使用它自己的认证配置。
   输出语言必须跟随用户当前请求语言；用户当前用中文提问时，最终结果必须输出中文。
   当用户当前请求语言很明确时，优先显式传入对应 `locale`；如果未显式传入，脚本会从当前会话里最近一条有实际文字内容的用户消息推断语言。
   如果输入来自聊天附件，先用 `node resolve-chat-images.ts` 取回临时文件路径，再把这 2 个路径传给 `recognize.ts`。
   不要改用模型原生视觉直接识别。

6. 只返回一次整合后的结果。
   不要按图片分别输出识别结果。
   如果 `node recognize.ts` 成功返回了最终识别文本，默认直接把该文本作为最终回复返回。
   不要改写、压缩、摘要、重排，也不要省略任何非空字段。
   只有在脚本返回错误信息、或用户明确要求另一种展示格式时，才允许你自己整理；即便整理，也必须保留脚本返回的全部非空字段。
   默认展示格式必须是纯文本逐行字段，例如 `名称: ...`、`年份: ...`。
   不要输出 Markdown 表格、`|` 管道对齐文本、围栏代码块、引用块或依赖等宽字体对齐的版式。
   特别是在 WhatsApp、Telegram 这类聊天渠道里，不要为了“更整齐”改成表格，因为这些渠道会把表格渲染得很奇怪。
   输出顺序上，把用户最关注的“币名/名称”和“估价”放在最前面。
   在结果结尾追加 1 行简短的收藏建议即可，不要展开成长段分析。

   结果中至少应保留这些核心字段：
   - 币名
   - 价格
   - 年份
   - 国家或地区
   - 面值
   如果脚本返回了这些补充字段，也必须一并保留：
   - Krause 编号
   - 材质
   - 直径
   - 厚度
   - 重量
   - 正面描述
   - 背面描述
   - 脚本实际返回的其它补充字段

   最终回复里的字段名不要附加来源说明。
   价格字段直接写“估价”或对应 locale 的正常字段名，不要写“估价（脚本）”“价格（脚本）”“来源：脚本”等标记。
   除非用户明确要求说明来源，否则不要解释哪些字段来自脚本、哪些字段来自整理。

   字段缺失时省略，不要臆造。

## 用户交互约束

- 用户一次发来 2 张图时，直接识别 1 次。
- 用户先发 1 张，后续补第 2 张时，在第 2 张到达后识别 1 次。
- assistant 明确催补图后，只有紧接着的下一条用户消息单独发 1 个显式图片 URL / 本地路径时，才自动把它并入当前未完成任务。
- 如果用户只是单独发 1 个显式图片 URL / 本地路径，且没有补图语义，也没有刚收到 assistant 的补图提示，默认把它当成新任务第 1 张图。
- 用户先连续发来 2 张聊天附件，随后再单独发一句“这是什么币？”时，直接用最近任务中的 2 张附件识别 1 次。
- 已经凑齐 2 张图后，不要再说“还差一张”。
- 不要在同一次任务中重复调用脚本。
- 用户上传超过 2 张图时，直接提示“`一次最多上传2张图，请重新上传`”，不要让用户再指定其中哪 2 张。
- 不要把当前消息中的多张图自动拆成多枚钱币，除非用户明确说是多枚钱币。
- 不要引用已完成旧任务里的图片或结果。
- 不要把历史未完成任务里的图片，和当前无补图语义的单张显式 URL / 本地路径自动拼成一次识别。
- 不要把脚本返回的完整结果改写成更短摘要，导致材质、直径、厚度、重量等非空字段丢失。
- 用户当前请求是中文时，不要输出英文结果；除非用户明确要求英文。
- 收藏建议保持 1 行、轻量、保守，不要扩展成投资建议或长篇判断。
- 默认不要使用 Markdown 表格、管道对齐、代码块或其它依赖富文本渲染的版式；最终回复应保持普通纯文本。
- 不要直接根据模型看图结果回答“这是什么币”。
